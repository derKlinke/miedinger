# Repository Guidelines

## Project Structure & Module Organization

- `src/` TypeScript CLI sources (entry logic, config generation, helpers).
- `dist/` built CLI output (`dist/format-configs.js`); generated by `npm run build`, commit when `src/` changes.
- `configs/` shared formatter config files shipped to consumers.
- `README.md`, `CHANGELOG.md`, `LICENSE` package docs and release notes.

## Build, Test, and Development Commands

Install deps (npm is the repo package manager):

```sh
npm install
```

Key scripts:

- `npm run build` TypeScript compile + adds CLI shebang and executable bit.
- `npm run format` Prettier + markdownlint over repo files.
- `npm run format:check` format + fails on diffs (CI-style check).

Run the CLI locally after build:

```sh
node dist/format-configs.js --help
```

## Coding Style & Naming Conventions

- Formatting: Prettier config at `configs/.prettierrc.json` (4-space indent, semicolons, double quotes, print width 100, LF). Markdown override uses 2-space indent.
- Linting: Markdownlint config at `configs/.markdownlint.json`.
- Naming: `camelCase` for variables/functions, `PascalCase` for types/classes, `kebab-case` for filenames (e.g., `format-configs.ts`).

## Testing Guidelines

- No automated tests currently. If you add business logic, add tests and a script entry (e.g., `npm run test`) and document it here.
- Until then, validate changes with `npm run build` and `npm run format:check`.

## Commit & Pull Request Guidelines

- Commit messages follow Conventional Commits (e.g., `feat: add sqlfluff preset`, `fix: handle empty config`).
- Commits that change config files must use `fix:` or `feat:` prefixes.
- PRs: include a short summary, linked issue(s) if applicable, and validation notes (commands run and results).
- If you modify `src/`, rebuild and include updated `dist/` artifacts in the PR.

## Release Notes

- Releases are automated via semantic-release; avoid manual version bumps in `package.json` or `CHANGELOG.md`.
